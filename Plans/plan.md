## PHYSICS

1. Implicit Free Lists for Branchless Sleep Management Checking if (slab.islandAsleep[i]) continue; thousands of times per frame inside hot loops destroys CPU pipeline efficiency through branch mispredictions. Instead of iterating over everything and skipping what is asleep, we can use an Implicit Free List (or array partitioning) to physically segregate memory. We maintain an activeConstraintCount pointer. When an island falls asleep, we execute a fast O(1) swap, moving its bodies and constraints to the end of the arrays and decrementing the active counter. The solver loops then simply become for (let i = 0; i < activeConstraintCount; i++). Sleeping objects mathematically cease to exist from the solver’s perspective—zero branching, zero iterations, and zero CPU cycles wasted on dormant physics.

2. Full Structure-of-Arrays (SoA) for Rigid Bodies While the kineticConstraintSlab beautifully applies Data-Oriented Design to constraints, it still resolves body velocities via object pointers (bodyA.vx, bodyB.vy). This forces the CPU to chase memory references across the heap, triggering L1 cache misses. The natural evolution is to flatten the entire active kinetic body pool into a unified Float32Array architecture (bodyVx, bodyVy, bodyInvMass). By resolving bodies entirely via integer indices inside the solver loop, we guarantee 100% linear memory access. Modern JavaScript JIT compilers (like V8) can aggressively unroll and vectorize these typed-array loops, easily yielding a 2x to 3x raw throughput multiplier for the core velocity solver.

3. Persistent Constraint Graphs & Manifold Caching Currently, the engine incurs a massive O(N) penalty every tick by completely tearing down and rebuilding the constraint slabs and wall candidate arrays. Physics engines fundamentally benefit from "temporal coherence"—objects mostly stay exactly where they were a millisecond ago. By upgrading to a Persistent Constraint Graph, the engine caches the topological network of contacts and joints. Instead of rebuilding arrays every frame, the system only patches the graph when a new collision begins, an old contact separates, or an island wakes up. This transforms the heaviest setup phases of the pipeline from a brute-force "recalculate everything" model into a lightweight "only update the delta" architecture.

4. Graph Coloring for Accelerated Solver Convergence Iterative constraint solvers process joints sequentially, meaning the array's order heavily biases the physical outcome (causing stretchy chains or uneven forces). By applying a greedy Graph Coloring algorithm to the constraint network during the gather phase, we can group constraints into independent "colors" where no two constraints share the same body. Solving all constraints of Color A, then Color B, mathematically guarantees zero memory contention between joints. This independence drastically accelerates solver convergence—meaning you get rock-solid, stiff physics in half the iterations. As a massive bonus, this colored independence perfectly sets up the physics engine to be multithreaded across Web Workers without race conditions.

5. Morton Codes and Hierarchical Bitsets for Broadphase The current grid broadphase relies on looping over AABBs and merging candidate arrays. We can revolutionize this spatial querying by mapping the 2D world grid into a 1D array using Morton Codes (Z-order curves), which mathematically guarantees that objects physically close in the world sit next to each other in RAM. If we back this up with a Hierarchical Bitset (a tree of 32-bit integers where a single bit represents grid chunk occupancy), broadphase culling becomes virtually instantaneous. Finding wall candidates or overlapping neighbors bypasses standard looping entirely, dropping down to raw CPU bitwise operations (&, |, and Math.clz32) to skip massive empty spaces in true O(1) time.

## PATHFINDING

**Goal of this series:** not new capability — _legibility_. Make the path from "set a target" to "a prop moves" travel through one obvious chain of ~4 modules, with one answer to "can I step here," one cache key that says "is nav stale," one place that owns shared memory, and one locomotion driver. No IGraph, navmesh, smoothing, or crowd work here — those are features that land _after_ the stack stops being a scavenger hunt. The six passes are ordered so each leans on the previous one: representation first, then the invalidation spine, then the worker boundary, then the agent layer, then the gameplay factors, then the lifecycle glue + tests that lock it in.

Part 1 — One nav-topology representation: the arena (worker bakes it for the live loop, in-process bake for batch authoring). Today walkability lives in two shapes: the `NavTopologySabArena` (`sabBlocked`, `sabCardinalOpen`, `sabVertexPassability`, `sabOctileNeighbors`, `sabOctilePredecessors` — a `SharedArrayBuffer` both threads read) and `GridNavContext`'s private plain `Uint8Array`s baked synchronously on main. The end-state of this pass is that the **arena is the single representation everywhere**, and `GridNavContext` as a distinct type dissolves into it. There is one set of pure bake functions (`recomputeVertexPassabilityInto`, `recomputeNavCardinalOpenInto`, octile/predecessor builders) — the single source of truth — invoked from two contexts that are inherent to the system, not duplication: the worker bakes the arena **off-thread for the live agent loop**, and batch authoring (procedural map-gen) + tests bake an arena **synchronously in-process**, because level generation interleaves grid edits and walkability queries thousands of times in one call and genuinely cannot round-trip a worker. Both query methods stay as pure reads over that one arena — `navCanStep` (octile CSR, used by the live loop) and `boundaryBlocksStepFrom` (`cardinalOpen`/`vertexPassability`, lighter, used by the generation loop so it never has to build octile per edit). One representation, one truth, one set of bake functions; the only variation is who triggers the bake.

**Already shipped (worker self-bake):**

- `HpaWorkerEntry.bakeNavTopology` now calls `recomputeVertexPassabilityInto` + `recomputeNavCardinalOpenInto` on its `navSimView` before building octile neighbors/predecessors — the worker produces the full topology (blocked → vertex → cardinal → octile → predecessors) in one place, off-thread.
- `HpaPathWorker.scheduleNavTopologySync` no longer calls `bakeNavCachesInto`; it ships only the raw grid/floor/edge SABs the worker needs. `bakeNavCachesInto` and its parity unit test are deleted (no remaining caller).

**Rest of this pass (the representation collapse):**

- `GridNavContext.js` — fold the type into the arena. `createGridNavContext` becomes "create a `NavTopologySabArena`," `syncGridNavContext` becomes "bake this grid into this arena" (the in-process batch path, calling the same pure functions the worker calls). Consumers read arena views, not a bespoke context object.
- `NavigationService` (live runtime) — stop self-baking. `onObstaclesChanged` keeps `flowFieldGrid.invalidateNavTopology()` and the async worker-sync chain, and drops the synchronous bake; its `gridNavContext` becomes a thin reader of the worker-baked arena. `gridCellVision` / `navWalkableCell` read that arena. Accepted tradeoff: perception is ~1 frame behind on a world edit — invisible for the rare belt/door/wall edit.
- Map-gen (`snakeSplitLayout.js`, `railMaze*`, `walkableCells.js`) — bake an arena in-process and query it via the pure functions; mechanical swap from "make a context, bake into its arrays" to "make an arena, bake into it." No move to the worker (batch authoring stays synchronous by necessity).
- Tests (~12) — adapt to the unified path: build + bake an arena synchronously (same code the worker runs), or run the real worker. `createTestNavigation` becomes a thin arena-baking helper, not a divergent synchronous `onObstaclesChanged`.
- `WorldObstacleGrid.canStep` — reads the arena. Live callers resolve to `navCanStep` against the worker arena; before the first sync it reports not-ready rather than baking (batch callers always have a freshly baked arena, so they're never in a not-ready window).

Consumers that already read the arena — `AStar.js`, the HPA region build via `navGraph.canStep`, the flow `neighborGrid` — need no change and inherit the unification, which is what removes the flow-vs-HPA drift at the root. (Explicitly out of scope: whether vision can later drop the precomputed array for per-query live checks — a measurable vision-perf follow-up, not part of this representation collapse.)

Part 2 — One invalidation spine (kill the epoch sprawl). There are ~11 staleness signals in flight: grid mutation channels (`wallGridRevision`, `floorNavEpoch`, `gridTopologyEpoch`, `_passagePowerNavKey`), the composite `gridNavCacheKey`, `NavigationService.obstacleGeneration`, per-entity `navState.obstacleGeneration`, worker `_graphEpoch`, host mirror `_navKey`, worker `navCacheKey`, and a `gridNavCacheKey` "synced" stamp on the grid. This pass documents and enforces a single spine: grid edits bump exactly one of the four channels (`bumpGridNavEpoch`), those fold into the one `gridNavCacheKey`, and _every_ downstream cache (the NavTopology arena — worker-baked for the live loop, in-process for authoring (Part 1) — the HPA region graph, the flow-field `_topologyKey`, and the per-entity replan epoch) keys off that single value — no module invents its own counter. Collapse the redundant mirrors (`_navKey`, worker `navCacheKey`, grid stamp) into reads of the one key, and write down for each surviving cache: what it stores, who bumps it, who reads it, and what clears it. The deliverable is that "is this nav data stale?" is answerable by comparing one number, anywhere in the stack.

Part 3 — One SAB + worker-protocol boundary (de-soup the threads). SharedArrayBuffers are currently allocated in four unrelated places — `hpaWorkerSab.js` (path/graph pools), `navTopologySab.js` (topology arena), `HpaPathWorker.js` inline (`sabEdgePool`), and `FlowFieldGrid.js` inline (flow pools) — and the layout constants are duplicated and drift-prone (`damagePadding=12` in both `HpaPathWorker.js` and `hpaRegionGraph.js`; path-length `512` redefined in corridor modules; slot strides re-derived per file). The worker protocol is equally ad-hoc: `syncObstacleNavGraph(grid, damageBounds, graphEpoch, seedWorldX, seedWorldY, fullGraph)` and `runOneShotReplan(slot, startCol, startRow, targetCol, targetRow, stepPenalty)` are positional arg-soup, and `init` nests its payload under `data` while every other message is flat on `e.data`. This pass gives nav one SAB-arena owner that allocates every shared buffer and exports all layout constants/offsets from one module, and one message-envelope contract (consistent shape, named fields, the structured-opts style that `requestPath(opts)` already uses) so a new message type can't reinvent the wrapping. It also fixes the rebind divergence the map surfaced — when `sabCellToRegionIdx`/`sabVertexPassability` are replaced, the worker must reliably re-bind on the next sync, not only on a full `rebindArena`.

Part 4 — One HPA locomotion driver (collapse the duplicated follow loop). `hpaGroundNavSession`, `hpaGroundNavBehavior`, `cellTargetHpaNav`, and `flowGroundNavBehavior` each carry near-identical update loops — epoch check → replan reason → idle/stuck replan → `computeSabPathSteering` → off-path replan — and the on-belt handoff block is _copy-pasted verbatim_ between `hpaGroundNavBehavior.js` and `cellTargetHpaNav.js` (with a third partial variant in `flowGroundNavBehavior.js`). This pass extracts one shared "target in → steering out" driver that owns the loop, and reduces the behaviors to thin adapters that differ only in target representation (world-click vs cell-destination vs belt-adjusted goal). The replan policy (`hpaReplanPolicy.js`) stays the single source of replan predicates, and _nothing_ calls `requestReplan` except the driver; `buildReplanParams` and endpoint snapping live with the request layer, not re-derived in each behavior. This is also where flow stops drifting: the flow behavior becomes an adapter over the same driver scaffold (sharing target snapping and belt handoff), differing only in that its steering backend samples the flow field instead of following SAB waypoints.

Part 5 — One source per world factor (belts & forcefields). Each gameplay factor is currently implemented across layers that don't share an API. Belts: wrong-way entry blocking lives in `boundaryOccupancy.js` (graph), goal-snapping exists in three variants (`snapPathTargetCell`, `resolveFloorBeltSteerTarget`, plus belt-cell arrival rules), on-belt handoff lives in locomotion, and the actual acceleration lives in `floorOccupancy.js` (physics) — and a single-cell belt edit bumps `floorNavEpoch` but often never calls `onObstaclesChanged`, so the worker graph silently misses it. Forcefields: the "is this powered edge passable?" question is answered separately in nav (`passageEdgeBlocksStep`), in physics collision (`passageEdgeBlocksCollision`/`passageWallContact`), and in render (`drawForcefields` power keys), all reading the same `edge.powered` flag through different code. This pass makes each factor own one nav-facing module: a belt module that owns belts' effect on planning (the one snapping helper the driver and the planner both call) and guarantees an edit triggers a resync, and a forcefield predicate that nav/collision/render all consult for passability so the three can't disagree. No change to belt _physics_ or forcefield _rendering_ pipelines — just one shared answer to the traversal question each one asks.

Part 6 — One frame pump + the safety net (lifecycle, dead-end deletion, tests). HPA replanning still depends on `beginFrame`/`flushFrame` being called from three places (editor `simulationFrameHooks`, snake perception, ad-hoc test calls), and the editor bug — hooks registered on full state but invoked on the slim `worldSimFromState()` `tick.world` that has no `sandbox`, so the optional chain no-ops and `flushFrame` never runs — shows how fragile that is. This pass introduces one explicit frame pump the sim loop calls each tick (resets the per-frame start budget, drains the replan queue) and routes editor, snake, and tests through it instead of duplicated wiring. It then deletes the dead ends the cleanup exposes: the legacy object-map `runAbstractAStar` (production uses the flat CSR version), any leftover single-request worker assumptions, and the now-removed divergent live-canStep path. Finally it adds the targeted tests that lock the legibility in: topology-bake parity (main vs worker produce identical `canStep`), session queue drain + supersede-on-new-target, apply-result → steer, belt handoff via the shared driver, and one-number epoch invalidation — all driven through the unified frame pump rather than per-test mock choreography. End state: set-target-to-prop-moves is one readable path, and the stack is ready for the actual feature work (smoothing, separation, navmesh) without first untangling it.

## UH?

Pass 1: Unifying Nav-Topology Representation (Folding GridNavContext into NavTopologySabArena)
To eliminate the dual walkability representations, we will fold the legacy GridNavContext type into the NavTopologySabArena. In navTopologySab.js, we will add structural aliases to the arena views—such as mapping arena.navCardinalOpen to arena.cardinalOpen and defining a dynamic wallRevision getter that reads directly from grid.wallGridRevision. This ensures that NavTopologySabArena is 100% structurally compatible with all existing code that currently expects a GridNavContext. We will refactor createGridNavContext to return a NavTopologySabArena, and update syncGridNavContext to perform synchronous, in-process baking of vertexPassability and cardinalOpen directly into the arena's views. This preserves the synchronous requirements of map-generation pipelines (like snakeSplitLayout.js and railMazeCorridorBelts.js) and unit tests without needing a worker thread.

In the live runtime, we will completely remove the synchronous syncGridNavContext call from NavigationService.onObstaclesChanged, eliminating redundant main-thread bakes during gameplay. Instead, NavigationService.gridNavContext will be assigned directly to the hpaPathWorker.\_navArena, acting as a thin, zero-cost reader of the worker-baked SharedArrayBuffer. WorldObstacleGrid.canStep will be simplified to read from this unified arena; live callers will automatically resolve to navCanStep against the worker-baked arena once the worker completes its sync, and before the first sync, it will report not-ready (false) rather than baking. This accepts a minor tradeoff where perception is up to one frame behind on a world edit, which is invisible for gameplay, while completely unifying the representation of walkability across threads.

Pass 2: Establishing One Invalidation Spine (Consolidating Epochs and Cache Keys)
To resolve the epoch sprawl and eliminate the ~11 different invalidation signals currently in flight, we will establish a single invalidation spine. Any grid modification (walls, floors, or topology) will bump exactly one of the three channels via bumpGridNavEpoch(grid, channel). These channels are aggregated into a single, canonical string key returned by gridNavCacheKey(grid). We will eliminate duplicate counters and mirrors across the codebase, removing stateful variables like \_navKey and worker navCacheKey in favor of direct comparisons against gridNavCacheKey(grid). The host mirror \_navKey and the grid's gridNavCacheKey stamp will be collapsed into direct reads of this single key, ensuring that "is this nav data stale?" is always answerable by comparing a single value anywhere in the stack.

Every downstream cache—including the NavTopologySabArena (worker-baked or in-process), the HPA region graph, the flow-field \_topologyKey, and the per-entity replan epoch—will key off this single gridNavCacheKey. We will write a clear, concise documentation block in gridNavEpoch.js that maps out the entire invalidation flow: what each channel represents, who bumps it, who reads it, and how cache validity is verified across the main thread and worker boundary. This guarantees that any edit to a single-cell belt, door, or wall reliably invalidates all downstream caches in lockstep, preventing the flow-field and HPA region graph from ever drifting out of sync.

## WORKER PLAN?

PR 1 — Kill the live double-bake. NavigationService.onObstaclesChanged currently sync-bakes into a private Uint8Array context and then async-syncs the worker — delete the syncGridNavContext call and stop allocating a separate main-thread context in the constructor. PR 3 — Map-gen on the worker, delete the bake module. snakeSplitLayout.js is the only procedural caller still sync-baking inline (syncGridNavContext mid-generation loop and the throwaway createGridNavContext path when no navigation is passed). ## REVISION

Here’s a revised 5-pass plan grounded in what’s actually landed (sync bake gone, worker is the bake site, tests and map-gen preview go through the worker) and focused on what’s still messy.

Pass 1 — One topology, one read API
Mostly done; finish the edges. Walkability now lives in one place (NavTopologySabArena, worker-baked). gridNavContext is just a live reader on that arena — good. What’s left is making the query surface obvious: navCanStep (octile, for movement/HPA/flow) vs boundaryBlocksStepFrom (cardinal/vertex, for belt-mouth and map-gen heuristics) should be documented as the two intentional reads over the same baked data, not two competing truths. Tighten WorldObstacleGrid.canStep so it always means “worker topology is ready and this step is open” (not-ready → false, no silent fallback). Optionally revisit map-gen: worker round-trips work but are slow; if preview/generation latency matters, add a local in-process arena bake using the same pure functions as HpaWorkerEntry.bakeNavTopology — same representation, different trigger, no second code path.

Pass 2 — One invalidation spine
Not started; highest leverage for “why is nav stale?” Grid edits still fan out into many signals (wallGridRevision, floorNavEpoch, gridTopologyEpoch, passage power, gridNavCacheKey, NavigationService.obstacleGeneration, per-entity epochs, worker \_navKey / \_graphEpoch, grid stamp). Collapse to a documented spine: edits bump channels → one canonical gridNavCacheKey(grid) → every downstream cache (topology arena, HPA region graph, flow-field topology, replan epoch) compares that one value. Delete redundant mirrors (\_navKey, stamped gridNavCacheKey on grid, duplicate worker keys) where they’re just copies. Outcome: debugging “nav is wrong” starts with one comparison, not a scavenger hunt across eleven counters.

Pass 3 — One worker boundary
SAB and protocol are still soup. Buffers are allocated in several modules with duplicated layout constants (damagePadding, path lengths, slot strides). Worker messages mix shapes (init nested under data, everything else flat; positional arg-soup on sync/replan). Give nav a single SAB owner (all pools + topology arena + layout constants/offsets exported from one module) and a single message envelope (named fields, structured opts like requestPath already does). Pass 4 — One locomotion driver
Agent layer is copy-pasted. hpaGroundNavSession, hpaGroundNavBehavior, cellTargetHpaNav, and flowGroundNavBehavior each carry nearly the same loop (epoch check → replan reason → stuck/idle → steering → off-path replan), with belt handoff duplicated verbatim. Extract one target-in → steering-out driver that owns that loop; behaviors become thin adapters (world click vs cell target vs flow sampling). Keep hpaReplanPolicy.js as the only replan predicate source; only the driver calls requestReplan. Pull belt goal-snapping into one helper the driver and planner share. Outcome: “prop doesn’t move” traces through ~4 modules, not four parallel implementations.

Pass 5 — One frame pump, one answer per world factor, tests that lock it in
Lifecycle glue and gameplay factors are still fragmented. HPA replan depends on beginFrame/flushFrame wired from editor hooks, snake perception, and ad-hoc tests — easy to no-op (editor worldSimFromState bug). Introduce one explicit per-tick frame pump the sim calls; route editor, snake, and tests through it. For belts and forcefields: one nav-facing module each — belt edits must reliably trigger resync; one passability predicate for powered edges that nav, physics, and render consult (they currently read the same flag through different code). Delete dead ends exposed by the cleanup (legacy A\*, divergent canStep paths). Add targeted tests: topology parity (worker vs in-process if you add batch bake), session queue drain, apply-result → steer, belt handoff via shared driver, single-key invalidation — all through the unified frame pump, not per-test mock wiring.

What you can drop from the old doc: the three-PR worker migration (done), GridNavContext as a type (gone), and the rigid “map-gen must never use a worker” rule (you chose worker for preview; Pass 1 leaves in-process as an optional perf path, not a requirement).

Suggested order: Pass 2 before Pass 3 (invalidation makes worker debugging sane), then Pass 4, then Pass 5. Pass 1 is mostly polish unless map-gen latency bites you.
