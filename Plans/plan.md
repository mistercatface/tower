PR 1 — View bounds tiers and sim visibility. Viewport already maintains five AABBs (boundsClip, boundsQuery, boundsDraw, boundsVisibleDefault, boundsVisibleWide), but callers pick them ad hoc and mix circle vs AABB tests. Render and most prop culling use boundsVisibleDefault; wall/rails collection uses boundsQuery; ground chunks use boundsDraw; editor overlays use boundsVisibleWide; animated surface zones use boundsClip via isAabbInView in floorShapes.js. Sim policy is separate again: nav uses isVisible(x, y, prop.radius), while brain sync uses (radius ?? 8) \* 2. The first PR should introduce a small tier API on Viewport — something like boundsFor("draw" | "query" | "visible" | "visibleWide" | "clip") plus intersectsCircle(ref, tier) and intersectsAabb(aabb, tier) — and migrate the sim/nav/AI call sites (isPropNavVisible, syncSpatialBrain, snakeGoals, Entity.isVisible) onto it. That makes “draw cull,” “sim wake,” and “editor margin” explicit instead of encoded in which raw field you grabbed. Behavior-preserving; no new abstractions beyond naming what already exists.

PR 2 — Unify entity-in-view queries through queryView. You currently have two prop-culling paths: entityRegistry.queryView (spatial broadphase + cached results, used for debris, 3D props, overlays, pick) and a hand-rolled worldProps loop in WorldSceneRenderer.drawFloorProps that does aabbOverlap(prop.aabb, bounds). The registry already supports configurable hit tests on queryInAabbStrict (center | circle), but queryView hardcodes "circle". Adding "aabb" to entityIntersectsAabb and exposing hitTest on QueryViewCriteria lets floor props join the same path as everything else — one broadphase, one cache key shape, one place to attach filterId/match predicates like renderMode === "floor". Extend the existing queryPropsInView helper (or a thin queryWorldPropsInView(viewport, spatialFrame, { tier, hitTest, match, filterId })) and migrate WorldSceneRenderer, buildSandboxOverlayCommands, and findWorldPropAtInView onto it. That removes the biggest structural duplication without touching grid stamps.

PR 3 — Grid-indexed and point cull helpers. Several hot paths convert a world AABB to grid cell ranges or do manual point-in-rect checks — duplicated verbatim in drawFloorOccupancyBelts and drawFloorOccupancyPowerSources, and again in spirit in collectForcefieldEdgeDrawables (comparing midX/midY against bounds edges instead of a proper overlap test) and roomGraphFloorDraw (correct aabbOverlap, but against a locally grabbed bounds). A single gridCellSpanFromBounds(grid, bounds) (returning clamped min/max col/row) and routing forcefield/room-graph culls through viewport.intersectsAabb(itemAabb, "visible") closes the remaining gaps. This PR is mostly mechanical, touches only draw paths, and pairs naturally with PR 1’s tier names so floorOccupancy stops reaching for viewport.boundsVisibleDefault directly.

That split is justified: PR 1 is foundational and unblocks the agent FSM work; PR 2 is the highest-value dedupe (eliminates a parallel culling loop); PR 3 is cleanup that depends on tier naming from PR 1 but is independently reviewable. What I would not add: a new query cache, a viewport service class, or wrapping every aabbOverlap call — extend Viewport, entityIntersectsAabb, and the existing queryPropsInView pattern. One thing to decide explicitly during PR 1: whether animated zones should stay on "clip" or move to "visible" — the tier API forces that choice into one documented line instead of hiding it in floorShapes.js.

## PR 1 FINAL

PR 1 collapses the five cached AABBs down to four named tiers with obvious meaning, deletes the duplicate, and rewires every caller that currently reaches for a raw field or the confusing isVisible(x, y, r, padding) overload. Today the rectangles are: boundsClip (0px — exact screen rect, editor map box + strict animated-zone cull), boundsVisibleDefault (20px — prop/entity cull for debris, 3D props, belts, forcefields, room patches, default queryView), boundsQuery (48px from viewQueryPadPx — wall/rail face collection so tall extruded geometry just off-screen still gets gathered), boundsDraw (128px from viewPaddingPx — ground-chunk bake/draw margin), and boundsVisibleWide (hardcoded 128px — used in exactly one place, buildSandboxOverlayCommands, so selection rings/path overlays still pick up props slightly off-screen). That last one is the dumb duplicate: boundsVisibleWide and boundsDraw are the same rectangle in production (both 128px), one driven by config and one by a stray constant — drop boundsVisibleWide and VIEWPORT_VISIBILITY_PAD_WIDE, and route overlays to the margin tier. Consolidation in Viewport.js becomes something like bounds("screen" | "cull" | "structure" | "margin") plus circleInBounds(x, y, r, tier) / aabbInBounds(aabb, tier); replace isVisible's padding-enum trick with tier args; fold intersectsAabb(aabb, 0) into aabbInBounds(aabb, "screen"). Touch list: Viewport.js (core), then every direct field reader — WorldSceneRenderer, StaticGridWallDraw, StaticGridEdgeRailDraw, wallDamageDraw, WorldSurfaceEngine, floorOccupancy, drawForcefields, roomGraphFloorDraw, sandboxOverlayCommands, buildSandboxOverlayCommands, floorShapes.isAabbInView, editor mapOverview / mapGenBoundsOverviewEditor — plus the sim-policy sites that should stop open-coding visibility: hpaGroundNavSession (isPropNavVisible), syncSpatialBrain, snakeGoals, and thin delegates like Entity.isVisible. Out of scope for PR 1: merging queryView with the floor-prop manual loop (PR 2) and grid cell-span helpers (PR 3); PR 1 is purely “what rectangle am I asking about?” and making sim vs draw vs editor margin explicit through tier names instead of five opaque fields.
