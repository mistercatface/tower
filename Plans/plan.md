Pass 1: Eliminate Canvas Cloning and Implement Object Pooling for Sprites The first PR will target the 28.1ms bottleneck in BakedSpriteCache.js's set method and the accompanying 45.3ms Major GC spikes. Currently, BakedSpriteCache unconditionally creates a copy of the sourceCanvas and issues an expensive drawImage call. This is completely redundant because QuantizedSpriteCache already allocates a fresh OffscreenCanvas during its bakeFn() execution. We will refactor BakedSpriteCache to take ownership of the provided canvas directly, instantly removing the double allocation. To further alleviate GC pressure and allocation overhead, we will introduce a canvas pooling mechanism in offscreenCanvas.js. When the LRU cache evicts an old sprite, its canvas will be returned to the pool and cleared (ctx.clearRect) rather than destroyed, allowing the next getOrBake call to recycle it. This cleanup will also significantly improve code readability by clarifying canvas lifecycle ownership.

Pass 2: Dynamic LRU Sizing via Cache-Pressure Telemetry The second PR will directly address the feedback regarding cache thrashing, which is the root cause of the high getOrBake (28.1ms) and base drawImage (97.3ms) times. Currently, propSpriteCache is hardcoded to a strict 2,560 maxItems limit. When a complex scene exceeds this threshold, the LRU begins evicting and re-baking sprites every frame. We will introduce cache-pressure telemetry that monitors cache miss rates and eviction frequencies. Based on this data, we will refactor createQuantizedSpriteCache to dynamically size its internal maxItems limit to fit the scene's unique sprite count, preventing thrashing entirely. Additionally, we will review and tighten the caching key logic (e.g., buildPropSpriteKey, quantizeViewOffset) to ensure we aren't inadvertently busting the cache with unnecessarily granular sub-pixel shifts, resulting in a cleaner, self-regulating caching architecture.

Pass 3: Optimize Canvas State Mutations and Affine Transformations The final PR will tackle the high-frequency execution costs inside drawImageWithModifier (34.4ms) and drawImageTriangle within AffineTexture.js (51.1ms total). In 2D canvas, context state changes are incredibly expensive. drawImageWithModifier currently wraps calls in save()/restore() or constantly mutates globalAlpha, while drawImageTriangle executes a heavy save(), beginPath(), clip(), transform(), and restore() sequence for every single polygon. We will optimize this by stripping out save() and restore() where manual state tracking can be used safely. For AffineTexture.js, we will attempt to bypass the clip() path entirely when drawing opaque geometries—perhaps by relying on geometric bleeding (bleedPx) instead—and swap relative transform() calls for absolute setTransform() calls. These changes will dramatically flatten the context mutation overhead and simplify the draw loop logic.
