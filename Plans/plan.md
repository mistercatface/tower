Part 1 — MazeLab becomes the full snake layout lab, not a rail-only toy. Rebuild MazeLab/ as a read-only preview over the same generators the game uses: upper band = V-CA cavern (generateCavernOccupancy / stampStaticWalls rules), middle = open padding strip, lower band = R-DFS rail maze (bakeRailMazeDfs + edge-rail topology). The lab does not fork maze math in a second copy; it imports from Libraries/Procedural/Mazes/ and Apps/Editor/world/mapWorld.js and renders with the same voxel vs railWall distinction documented in Plans/Mazes.md (filled cells vs boundary edges). Canvas layers: voxels, rail walls, north reserve, walkable flood — all driven by seed, play-area size, and the same config knobs as Config/games/snake.js. Success criterion: what you see in MazeLab at 256×256 is the same geometry the game would stamp, just drawn in 2D instead of through world surfaces.

Part 2 — Belts use the room-graph corridor belt rules, including elbows. Drop the straight-run-only planner that caused down-into-left dead ends. Belt candidacy comes from path polylines on the stamped walk grid (corridor runs between junctions/dead-ends), and each cell gets the same kind/facing resolution as roomGraphCorridorBelts.js: straight BeltRails, turns as BeltElbowLeftRails / BeltElbowRightRails via floorBeltEntryExitSides / resolveRailedBeltFromSides, with chain validation like assertBeltChains in tests/corridorHarness.js (entry exit of upstream matches downstream entry; no isolated dead-end belt cells). MazeLab draws belt arrows and elbow glyphs so bad topology is visible before anything touches the game. Planning stays pure data (arrays of { col, row, kind, facingIndex }); no WorldObstacleGrid, no nav worker, no HPA during preview.

Part 3 — One layout blob, one apply pass, no performance theater. MazeLab’s “Generate” output is a single SnakeLayoutPlan (or reuse sandboxSceneSnapshot fields: voxels, railWalls, floorBelts, bounds, seed) produced entirely in memory. The game path is: generateSnakeSplitMap → call shared bakeSnakeLayoutPlan(seed, bounds, config) → applySnakeLayoutPlanQuiet(state, plan) which batch-stamps voxels, batch-stamps rails (stampGlobalRailWalls), batch-stamps belts (stampBakedFloorBeltsQuiet), unions damageBounds, then one publishLocalNavAfterObstacleEdit + background scheduleWorkerNavSync — no commitBoundaryEdit nav storms, no per-chunk onObstaclesChanged, no belt validation loop that rebuilds the graph thousands of times. Targeted tests: MazeLab plan matches game plan for fixed seeds; belt harness passes on rail-maze paths; 256×256 full bake under a few seconds in Node with zero trap cells. Only after MazeLab is trustworthy do we wire snake mapgen to consume that single plan API.
