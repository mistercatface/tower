Next set of three PRs (engine + sandbox payoff). I’d end this set with (1) constraint infrastructure — joint data on props, anchor frames, post‑contact constraint solve, debug draw; (2) linked/island policy — wake/sleep and broadphase grouping so a chain isn’t N independent sleep vetoers, plus circle–circle cheap overlap at pair gather; (3) sandbox: ball chain / snake — place links in editor, head prop accepts nav/steering, tail follows, stress test becomes “long chain through cavern” instead of 500 independent balls. Endgame: dogpile case gets cheaper when links reduce effective independent contacts, and you have a feature players can see.

PR 1 — Constraint infrastructure. Add a real constraint layer that runs after contact solve, separate from the pair stream you just built. Joints live on props as data: type (start with distance), body A/B ids, anchor points in each body’s local frame, rest length, stiffness/compliance, optional break force. The solver pass applies positional corrections and optionally velocity-level impulses to satisfy those constraints for one or a few iterations — same warm-start/cache patterns as contacts, but a different buffer and a different loop. Wire debug draw through the overlay pipeline (anchor gizmos, link lines, tension color) so you can see constraints working before any gameplay depends on them. Success criteria: two unlinked balls still behave exactly as today; two balls with a distance joint stay at fixed separation under gravity and collision; no joint logic leaks into gatherKineticCandidatePairs or fracture paths.

PR 2 — Linked/island policy + cheap circle pairs. Treat linked bodies as a connectivity island for sleep and wake: if the head moves or gets steered, wake the island once; if the whole island is at rest, sleep it together instead of each link vetoing its neighbor forever. Broadphase can skip internal island pairs entirely (linked bodies don’t need contact resolution between each other — the constraint handles that), which directly cuts pair count for chains and is a big win even in non-snake scenarios. At pair gather, when both bodies are circle-tier and snapshotted as circles, use a direct center-distance overlap check instead of routing through the full pairBroadphaseBoundsOverlap / OBB path — cheap filter before the existing snapshotted policy. Success criteria: a 20-ball chain at rest sleeps as one unit; the same chain linked internally generates far fewer candidate pairs than 20 free balls; dogpile-with-links profile shows gatherKineticCandidatePairs and overlap cost drop relative to 500 independent balls.

PR 3 — Sandbox: ball chain / snake. Ship the visible feature on top of PR 1–2: a link tool or prop wiring in the editor (select head, link to next segment, repeat), distance joints between consecutive balls, head marked as chainHead (or similar) and wired to existing ground nav / steering so only the front drives intent. Tail segments follow via constraints, not friction stacking. Add a stress-test variant: spawn a long chain through the cavern instead of 500 loose balls — same benchmark slot, but it demonstrates the engine work and stays interactive (pan, nav head, watch tail whip). Success criteria: place and edit a chain in sandbox, steer head to a point, tail follows without exploding; internal links don’t double-resolve as contacts; perf is acceptable for chain lengths you’d actually use in puzzles (20–50 segments).

The set after that (second trilogy). Aim for manifold persistence + iteration budget — warm pair/manifold cache, optional early‑out when awake set and pair stream are stable across substeps, and a dedicated circle–circle solver path in the impulse phase (not just narrow phase). Middle PR: revolute + motor joints (pin hinge, flipper‑class powered rotation, limits). Capstone sandbox: mixed‑shape chains or ragdolls — poly links, breakable links (tie into existing fracture), maybe “train” of crates. That moves you from “fast pile of balls” toward something that feels like a real sandbox physics toy while keeping the pipeline you just built: snapshot → pair stream → tiered narrow phase → contact solve → constraint solve.

Wall overlap / chain vs buildings. What you’re seeing is expected with the current design: only the head gets HPA, constraints pull the tail through the head’s path, and nothing checks the full chain envelope against static walls — so a long chain can “cut corners” and clip voxels when the head takes a tight turn or a segment lags behind. A useful test would be an integration-style check, not a unit test on one ball: spawn a multi-segment chain, issue nav to several waypoints (or replay recorded head motion), step physics for N frames, and assert every segment’s broadphase circle (or anchor) never overlaps blocked cells / never has center inside wall occupancy. That catches tunneling and overlap regressions without requiring perfect gameplay. It won’t fix the behavior by itself, but it documents the gap and guards future work (chain-aware nav, swept-volume checks, or segment wall resolve).

Did we complete PR 3? Yes, for the stated success criteria: editor link tool, chainHead metadata, nav/steering only on the head (and path overlay filtered accordingly), distance joints on commit, stress demo as a cavern chain instead of 500 loose balls, internal links skipped at pair gather (PR 2), constraint + debug overlays (PR 1), and island wake/sleep (PR 2). Loose ends that are real but out of scope for “PR 3 done”: constraints and chainHead are not in scene snapshot export/import (refresh or load demo is fine; copy/paste layout is not); PR 1 didn’t ship stiffness/compliance/break force; the stress chain path is greedy cell-walk, not “wrap cleanly around geometry”; and we fixed island wake adding evicted bodies to the active set, which was a genuine PR 2/3 interaction bug. Cleanup opportunities without big new features: persist kineticConstraints + chainHead at save boundaries if you care about editor workflows; optional chain-path spawn that biases away from wall cells; no obvious dead-code purge needed from this set — the new modules are all wired. Easy perf wins already landed (skip internal island pairs, circle-tier overlap at gather); further wins are mostly second-trilogy (manifold cache, stable substep early-out), not tweaks to PR 3.

Second trilogy — still a good goal? Yes. The first set gave you the pipeline shape (snapshot → pairs → tiered narrow phase → contact → constraints) and a visible sandbox feature; the second set is what makes it feel like a “real” engine rather than a fast ball pile: manifold persistence and substep stability reduce redundant work when the awake set isn’t changing, a dedicated circle–circle impulse path completes what PR 3 started in narrow phase only, revolute/motor joints unlock flippers and powered hinges without special cases, and mixed-shape / breakable links connect to fracture and crate trains. That’s the right split between professional physics and game-facing toys — each PR stays testable and each capstone is something you can see in the sandbox.

Snake autosim sideshow. A self-running snake scene is a strong fit for what you already built and doesn’t need the second trilogy first: spawn a head + two linked nav balls (or a dedicated snake head asset), mark head as chainHead, place a food prop, run HPA on the head toward food each time food moves, and when head is within eat radius remove/despawn food, addDistanceConstraint a new segment at the tail (rest length = spacing), respawn food at a random open cavern cell, repeat. No player input — just a little autonomous loop that stress-tests constraints, island wake, head-only nav, and growth in one scene. It’s narrower than full editor snake tooling but validates the chain stack in a way the cavern whip test doesn’t (persistent growth, repeated link creation, goal-seeking). Good warm-up before revolute joints and breakable links in trilogy two.
