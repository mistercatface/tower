PR 1 — Multi-snake gameplay + per-snake random color (game layer only)

Refactor snake setup from a single chain/autosim into a small session that owns an array of snakes: player snake (camera follows, HUD tracks this one) plus one rival at 3 segments with its own createSnakeAutosim, no camera target. spawnSnakeCavernScene picks two non-overlapping anchor cells (exclude occupied cells after each placement) and gives each snake its own goal orb and goal lifecycle — same rules, independent orbs. For color, pick a random hue per snake at spawn and stamp the same generated panel palette on head + all body segments via a lightweight prop.spherePanels field; extend spherePrimitive to read it and add a cache-key suffix so tints rebake correctly. New segments on eat inherit the chain palette. Neutralize snake_head base panels (grey, not green) so tint reads clearly. Config gets rivalCount: 1 (or snakes: [{ segments, cameraFollow, faction }]) rather than hardcoding. Tests: scene spawns two chains, player-only camera, grow preserves tint, two autosims don't stomp each other's goals.

While you're in there: multiple concurrent HPA ground-nav seekers on one grid is a good stress test for the nav session and worker queue — if replans contend or goals go stale after the other snake eats, tighten goalSeekAutosim so a dead goal triggers a shared respawn hook or each autosim's getGoalPropId stays live. Chain growth should keep using resyncChainLinkRestLengths after radius steps so two long chains don't drift link rest lengths. No new physics types needed; the needle-mover is validating multi-agent nav + constraint chains under real overlap.

PR 2 — Sandbox prop tint (color picker) + snake on the shared API

Replace PR 1's ad-hoc spherePanels with a first-class visual override: propTint on WorldProp, snapshot schema bump (optional tint on flat prop entries), spawn-default tint in the place palette inspector, and selected-prop color picker for tintable assets (tag "shapes" sphere primitives to start). Rendering goes through a small Libraries/Render/propTint.js — hue-shift or recolor asset base panels, bucket in getCustomSpriteCacheKey, same path as pool-ball instance keys. Snake spawn calls setPropTint(prop, hue) on chain members; editor users get the same control when placing balls or heads manually. Optionally show the spawn swatch from resolved tint, not just asset default.

This PR is mostly render + editor + persistence, but it pays down prop-catalog duplication (orange_ball, blue_ball, …) and sets up instance variation for any future autosim faction colors. If you extend tint to extruded props (crates, goal star), the bake pipeline stays one path through cached props — no fifth draw pipeline. Physics-wise the win is indirect: fewer one-off assets means consistent mass/radius/friction definitions and less drift between "same ball, different color" variants.
